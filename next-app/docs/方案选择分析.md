# 测试案例管理系统 - 技术方案选择分析

## 📋 需求背景

将现有的测试案例管理系统（Alpine.js + Tailwind CSS）迁移到 Next.js + React + Ant Design 项目，并复用现有的后端 API 服务（`/src/http-server/test-case-api.js`）。

## 🎯 候选方案

### 方案 1: Next.js 代理方式 ⭐（推荐）

**实现思路**：
- 利用 Next.js 的 `rewrites` 配置
- 开发环境下将 `/api/*` 请求代理到后端服务器（端口 3000）
- 前端代码使用相对路径 `/api/test-cases`

**架构图**：
```
┌─────────────────┐
│  Browser        │
│  localhost:3030 │
└────────┬────────┘
         │
         │ Fetch /api/test-cases
         ↓
┌─────────────────┐
│  Next.js Server │  (Rewrites 配置)
│  localhost:3030 │
└────────┬────────┘
         │
         │ Proxy /api/*
         ↓
┌─────────────────┐
│ Express Server  │
│ localhost:3000  │
└─────────────────┘
```

**实现代码**：
```javascript
// next.config.mjs
async rewrites() {
  if (isDev) {
    return [
      {
        source: '/api/:path*',
        destination: 'http://localhost:3000/api/:path*',
      },
    ];
  }
  return [];
}
```

**开发成本**：
- ✅ 配置修改：10 分钟
- ✅ 代码调整：0 分钟（无需修改）
- ✅ 测试验证：30 分钟
- ✅ **总计：约 40 分钟**

### 方案 2: API Routes 集成方式

**实现思路**：
- 在 Next.js 中创建 API Routes（`pages/api/test-cases/[...path].ts`）
- 直接导入并使用现有的 `test-case-api.js` 和 `TestCase.js`
- 将 @yanit/jsondb 数据库迁移到 Next.js 项目

**架构图**：
```
┌─────────────────┐
│  Browser        │
│  localhost:3030 │
└────────┬────────┘
         │
         │ Fetch /api/test-cases
         ↓
┌─────────────────┐
│ Next.js API     │  (pages/api/test-cases/)
│ Routes          │
│ localhost:3030  │
└────────┬────────
         │
         │ Import
         ↓
┌─────────────────┐
│ test-case-api.js│  (复用现有代码)
└────────────────┘
         │
         ↓
┌─────────────────┐
│ TestCase.js     │  (Model 层)
└────────┬────────
         │
         ↓
┌─────────────────┐
│ .jsondb         │  (数据库文件)
└─────────────────┘
```

**实现代码**：
```typescript
// pages/api/test-cases/[...path].ts
import express from 'express';
import testCaseApiRouter from '../../../src/http-server/test-case-api';

const app = express();

// 需要处理 Express 中间件与 Next.js API 的兼容性问题
app.use('/api/test-cases', testCaseApiRouter);

export default async function handler(req, res) {
  // 将 Next.js 请求转发给 Express 处理
  // 实现复杂，需要处理 Stream、Headers 等
}
```

**开发成本**：
- ⚠️ API Route 创建：1 小时
- ⚠️ 路径调整：2 小时
- ⚠️ 依赖安装和配置：30 分钟
- ⚠️ ESM/CJS 兼容性：1 小时
- ⚠️ 测试验证：1 小时
- ⚠️ **总计：约 5.5 小时**

## ✅ 最终选择：方案 1（代理方式）

### 选择理由

#### 1. **成本优势明显** ⭐⭐⭐⭐⭐
- 节省约 90% 的开发时间（40 分钟 vs 5.5 小时）
- 几乎零代码改动，降低引入 bug 的风险
- 配置简单，易于理解和维护

#### 2. **架构最佳实践** ⭐⭐⭐⭐⭐
- 前后端分离，职责清晰
- 符合现代 Web 开发标准
- 便于团队协作和代码审查

#### 3. **现有代码已就绪** ⭐⭐⭐⭐⭐
- 所有前端组件已使用相对路径 `/api/test-cases`
- 无需修改任何业务逻辑代码
- 组件、类型定义、请求封装都已完善

#### 4. **灵活性和扩展性** ⭐⭐⭐⭐⭐
- 后端 API 可被多个前端复用（H5、小程序等）
- 易于水平扩展（API 独立部署）
- 支持微服务架构演进

#### 5. **生产部署简单** ⭐⭐⭐⭐⭐
- Next.js 构建为静态文件
- 可使用 CDN 加速
- 只需配置 Nginx 反向代理

## 📝 关键决策点

### 为什么选择代理而不是 API Routes？

1. **时间成本**：40 分钟 vs 5.5 小时，节省 90% 时间
2. **技术风险**：零代码改动 vs 大量重构
3. **架构清晰**：前后端分离 vs 混合架构
4. **可维护性**：职责明确 vs 代码耦合

### 为什么接受需要启动两个服务？

1. **开发体验影响小**：一个脚本自动启动，无需手动操作
2. **架构收益大**：保持前后端分离，便于扩展
3. **行业标准**：大厂普遍采用此架构
4. **生产部署简单**：静态文件 + CDN，成本低

### 为什么选择 Next.js Pages Router 而不是 App Router？

1. **项目一致性**：当前项目使用 Pages Router
2. **稳定性**：Pages Router 更成熟稳定
3. **学习曲线**：团队更熟悉 Pages Router
4. **Ant Design 兼容**：无 SSR 水合问题
