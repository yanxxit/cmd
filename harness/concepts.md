# 常用 Harness 概念

## 1. Requirement
描述本次任务要解决什么问题、目标用户是谁、边界是什么。

## 2. Design
把目标转成结构化方案，至少说明：影响范围、目录落点、验证策略。

## 3. Tasks
将设计拆成可执行条目，通常放在 `tasks/` 或模块自带的 tasks 文档。

## 4. TDD
对于代码改动，优先先补最小测试，再实施功能；对于文档与配置改动，可以以脚本校验代替新增测试，但需要在交付说明中解释。

## 5. Verify
验证不等于只跑单一命令。至少要说明：
- 执行了什么脚本或测试
- 覆盖了哪些风险点
- 有哪些未覆盖风险

## 6. Decision Record
当仓库级约束发生变化，例如目录约定、验证入口、技能模板结构，需要在 `docs/decisions/` 中留下 ADR。

## 7. Skill Pack
仓库内自定义 skill 的最小集合建议包含：
- `SKILL.md`：作用与触发条件
- `FLOW.md`：执行步骤
- `TEMPLATES.md`：标准输出模板

## 8. Project Context
项目上下文不是长篇背景资料，而是帮助代理快速理解仓库现状的摘要，应由脚本生成，避免手写漂移。
